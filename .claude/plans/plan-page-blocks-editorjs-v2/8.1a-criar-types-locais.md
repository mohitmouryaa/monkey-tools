✅ # 8.1a — Criar tipos locais do `editor-js-field`

## Contexto

`EditorJSField` precisa de uns poucos tipos compartilhados entre o wrapper e os 6 form-modals: o shape de `EditingBlock` (qual bloco está sendo editado + callback de save), as props comuns dos modais (open/close, dados iniciais, callback). Concentrar em um arquivo pequeno evita repetição e ciclo de import.

## Detalhe técnico

Criar `apps/web/modules/dashboard/ui/components/editor-js-field/types.ts`. Conteúdo:

```ts
import type {
  CardsBlockData,
  CtaBlockData,
  FaqBlockData,
  HeroBlockData,
  RawHtmlBlockData,
  StepsBlockData,
} from "@workspace/types";

export type EditableBlockType = "hero" | "steps" | "cards" | "faq" | "cta" | "raw-html";

// Mapa do tipo do bloco para o shape do `data` correspondente.
export type BlockDataByType = {
  hero: HeroBlockData;
  steps: StepsBlockData;
  cards: CardsBlockData;
  faq: FaqBlockData;
  cta: CtaBlockData;
  "raw-html": RawHtmlBlockData;
};

// Estado de edição. `T` é estreitado pelo `type` em runtime via switch.
export interface EditingBlock<T extends EditableBlockType = EditableBlockType> {
  type: T;
  data: BlockDataByType[T];
  onSave: (next: BlockDataByType[T]) => void;
}

// Props compartilhadas pelos 6 form-modals.
export interface BlockFormModalProps<T extends EditableBlockType> {
  open: boolean;
  initialData: BlockDataByType[T];
  onCancel: () => void;
  onSave: (next: BlockDataByType[T]) => void;
}
```

**Importante:** importar tipos de `@workspace/types` (re-export já existente em `packages/types/src/index.ts` — verificado no plano). Não criar tipos locais que dupliquem os de `@workspace/types`.

- **P4** (A — `value` opcional): não afeta este arquivo (tipos puros).
