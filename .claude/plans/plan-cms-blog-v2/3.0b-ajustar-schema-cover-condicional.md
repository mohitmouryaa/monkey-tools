# ✅ 3.0b — Ajustar schema do post (`coverImage` condicional)

## Contexto
Decisão P6: `coverImage` é obrigatório apenas quando `status` ∈ `{SCHEDULED, PUBLISHED}`. Em `DRAFT` aceita vazio (admin pode salvar rascunho incompleto).

## Detalhe técnico
Em `apps/web/modules/dashboard/schema/post.tsx`:

1. Relaxar `coverImage` para `z.string().default("")` no objeto base (remover `min(1)`).
2. Aplicar `superRefine` no `createPostSchema` para emitir issue em `coverImage` quando `status` é `SCHEDULED` ou `PUBLISHED` e `coverImage` estiver vazio.
3. `updatePostSchema` herda automaticamente do `createPostSchema.extend(...)`. **Verificar** se o `extend` preserva o `superRefine`. Em Zod v3 padrão, refinements são preservados em `.extend`. Se houver algum problema de tipagem, replicar o `superRefine` no `updatePostSchema`.

Exemplo do bloco refine:

```ts
.superRefine((data, ctx) => {
  if (
    (data.status === PostStatus.SCHEDULED || data.status === PostStatus.PUBLISHED) &&
    (!data.coverImage || data.coverImage.trim() === "")
  ) {
    ctx.addIssue({
      code: z.ZodIssueCode.custom,
      message: "Cover image is required when scheduling or publishing",
      path: ["coverImage"],
    });
  }
});
```

## Arquivos tocados
- `apps/web/modules/dashboard/schema/post.tsx` — relaxar `coverImage` e adicionar `superRefine`.

## Verificação
- [ ] `pnpm --filter web typecheck` continua passando.
- [ ] Em `DRAFT`, `createPostSchema.parse({ ..., coverImage: "" })` não lança.
- [ ] Em `PUBLISHED`, `createPostSchema.parse({ ..., coverImage: "" })` lança erro com path `["coverImage"]`.

## Decisões aplicadas
- **P6 (A)**: `superRefine` aplicado para validação condicional.

## Atenção
A Fase 2 do `postsRouter` valida via `createPostSchema`. Esta mudança relaxa a validação do servidor também, o que é o comportamento desejado (DRAFT pode ser salvo sem cover). Não há ajuste extra no `postsRouter`.
